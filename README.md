# MailACL

MailACL generates deterministic, authenticated email aliases such as:

```text
service.example-<tag>@example.com
```

Each identifier gets a reproducible HMAC-SHA-512 tag derived locally from secret GPG key material. This makes aliases useful for identifying where an address was shared and for rejecting randomly invented aliases without maintaining an issuance database.

MailACL does **not** sign email messages, prove who sent a message, prevent forwarding, or configure a mail server. Its tags are truncated authenticators: resistance to guessing depends on tag length and on keeping the GPG secret key private.

## Features

- Deterministic aliases: the same secret key and identifier produce the same tag.
- Default lowercase Base36 tags with a 10-character suffix (about 51.7 bits).
- Configurable tag lengths, including intentionally weak short tags for restrictive forms.
- Verification of current Base36 suffix tags.
- Compatibility verification for earlier Base36-prefix, Base32, hexadecimal, and exact-case Base64 tags.
- RFC 5321 64-byte local-part enforcement during generation and verification.
- Optional high-contrast terminal output; canonical address lines remain plain text.
- Dependency-light Bash tests that use a synthetic fake `gpg`, never a real key.

## Requirements

- Bash 4 or newer
- GnuPG (`gpg`)
- Python 3
- Git, if cloning the repository

On Debian or Ubuntu:

```bash
sudo apt update
sudo apt install git gnupg python3
```

OpenSSL is not required.

```bash
git clone https://github.com/robertlanghorn/mailacl.git
cd mailacl
chmod +x mailacl.sh
```

## Configuration

Choose a GPG secret key and the domain where aliases are received:

```bash
gpg --list-secret-keys --keyid-format LONG
export MAILACL_GPG_KEY="YOUR_GPG_KEY_ID"
export MAILACL_DOMAIN="example.com"
```

Use a dedicated, backed-up key when practical. Editing, replacing, or losing the key can change the exported key bytes and therefore change generated tags.

Optional settings:

```bash
export MAILACL_SIGNATURE_ENCODING=base36  # base36, base32, base64, or hex
export MAILACL_COLOR=auto                 # auto, always, or never
```

`MAILACL_ENCODING` remains accepted as a legacy fallback for the encoding setting. `NO_COLOR` disables color when `MAILACL_COLOR=auto`.

## Usage

Show command help without loading a key:

```bash
./mailacl.sh --help
```

Generate with the default 10-character Base36 tag:

```bash
./mailacl.sh service.example
```

Choose a tag length:

```bash
./mailacl.sh service.example 16
./mailacl.sh service.example max
```

Verify an address:

```bash
./mailacl.sh 'service.example-<tag>@example.com'
```

Run without arguments for interactive prompts:

```bash
./mailacl.sh
```

### Suggested identifiers

Use an underscore for a person's name and the site's hostname for a website or service:

```text
Person:  john_doe-<tag>@example.com
Website: google.com-<tag>@example.com
```

For example, generate those aliases with:

```bash
./mailacl.sh john_doe
./mailacl.sh google.com
```

This convention keeps labels readable when an alias later appears in logs or a mailbox. Identifiers use unquoted ASCII email local-part (`dot-atom`) characters; spaces, `@`, non-ASCII text, and leading, trailing, or consecutive dots are rejected. The `<tag>` placeholder is generated by MailACL; do not type it literally.

Exit codes are `0` for success, `1` for configuration or verification failure, and `2` for command-line usage errors.

## Tag format and shortening

New aliases use lowercase Base36 (`0-9a-z`) by default. A 10-character suffix has approximately 51.7 effective bits. MailACL reports a digest-capped effective-entropy estimate, a numerically stable one-million-guess probability, and a strength label for current tags and the standard compatibility encodings. It deliberately does not assign a uniform entropy estimate to transitional fixed-width Base36 prefixes because their leading characters are biased.

Base36 tags are taken from the **right-hand end** of a fixed-width encoding. To shorten one manually, remove characters immediately after the final hyphen and keep the characters nearest `@`:

```text
Longer:  service.example-abcdefghij@example.com
Shorter: service.example-cdefghij@example.com
```

The strings above illustrate placement only; they are not valid tags for a real key.

Very short tags are intentionally supported for forms with strict address limits, but they are easy to guess. A one-character Base36 tag has only about 5.17 bits of search space. Treat short tags as labels or typo checks, not strong authenticators.

Verification remains compatible with historical formats:

- current Base36 suffix tags, case-normalized;
- transitional Base36 prefix tags, case-normalized;
- transitional Base32 tags, case-normalized;
- legacy hexadecimal tags, case-normalized;
- transitional standard Base64 tags, matched exactly because Base64 is case-sensitive.

## Security model and limitations

MailACL hashes the configured GPG secret-key export with SHA-512 and uses the resulting ASCII hexadecimal digest as the HMAC key. This unusual derivation is retained for compatibility with already-issued aliases. The HMAC key is sent to Python over standard input rather than placed in process arguments.

A matching tag means that the identifier agrees with the configured secret-derived key to the truncated tag length. It does not establish sender identity, message integrity, delivery history, or exclusive ownership of the identifier. Anyone who obtains the GPG secret key or derived HMAC key can generate valid aliases. Anyone can also guess a valid short tag with probability determined by its length.

The domain is validated and checked during verification, but it is not part of the HMAC input. The authenticated input is the identifier before the final hyphen.

For the complete algorithm and compatibility rules, see [theory_of_operation.md](theory_of_operation.md). To report a vulnerability, see [SECURITY.md](SECURITY.md).

## Testing

The regression suite creates an isolated fake `gpg` executable and uses public synthetic fixture bytes:

```bash
bash -n mailacl.sh tests/test_mailacl.sh
bash tests/test_mailacl.sh
```

If ShellCheck is installed, run the same static check used by CI:

```bash
shellcheck mailacl.sh tests/test_mailacl.sh
```

The tests cover fixed compatibility vectors, truncation, argument handling, byte-oriented local-part limits, domain and identifier validation, entropy/probability reporting, TTY and non-TTY color controls, plain canonical output, failed key export, host-environment isolation, and the no-OpenSSL implementation.
